# ScalarDBユースケースまとめに含めるべき情報

ScalarDBは複数種類のデータベースを統合し、トランザクションと分析処理を両立するユニバーサルHTAPエンジンです。公式ドキュメントでは主なユースケースとして「異なるデータベース間のデータサイロ解消」「マイクロサービス間の整合性確保」「データメッシュの管理簡素化」「データベース移行のハードル削減」を挙げています。

参考: [scalardb.scalar-labs.com](https://scalardb.scalar-labs.com)

製品ページでは、分散環境のデータを統一的に扱えることや、複数のデータベース間でACIDトランザクションとリアルタイム分析を実現できること、認証・暗号化・細粒度アクセス制御などの一元的なセキュリティ機能、ベンダーロックイン回避、マイクロサービスへの適用、RAG/AIのためのリアルタイム分析などが強調されています。これらの特徴を踏まえ、顧客の課題をヒアリングした際に適切な提案ができるよう、ユースケースを以下のような情報の塊（カテゴリ）で整理すると良いでしょう。

## 各ユースケースで整理すべき主な情報

| カテゴリ | 収集するキーワード・フレーズ（長い説明は表外で） |
|---------|------------------------------------------------|
| **ユースケース名称・概要** | ユースケース名と1〜2行程度の概要。扱う業界やビジネス領域（製造、金融、放送、エネルギーなど）を含める。 |
| **顧客のビジネス課題** | 顧客が解決したい課題や痛点。例えば「データサイロによる分析の遅延」「GHG排出量の正確な集計と報告」「メインフレーム維持コストの高騰」など。 |
| **システム／技術課題** | 既存システム構成やデータベースが抱える技術的制約。例えば「複数データベース間のトランザクション管理が複雑」「COBOLシステムとクラウドの連携が困難」「ベンダーロックインによりDB移行ができない」など。 |
| **対象データ・対象システム** | 対象となるデータの種類（文書ファイル、IoTデータ、CRMデータ、同意履歴など）やシステムの種類（RDBMS、NoSQL、メインフレーム、クラウドストレージなど）。 |
| **導入の目的／期待する効果** | ユースケースで達成したい目的や導入後に期待される効果。「データ整合性の確保」「リアルタイム分析による意思決定スピード向上」「COBOL資産の活用と段階的なクラウド移行」「RAG/AI用データ基盤の整備」など。 |
| **解決方法・ソリューション概要** | ScalarDBのどの機能を用いるか、他製品（ScalarDL、AI/LLM、Vector Storeなど）を組み合わせるか。例えば「ScalarDBで複数DB間のACIDトランザクションを実現」「ScalarDBの統一インタフェースでポリグロット・パーシステンスを実現」「ScalarDBとScalarDLでファイルの非改ざん性を保証」など。 |
| **システム構成／アーキテクチャ** | ユースケース実現のための構成図やデプロイパターン。サービス・アプリと各データベースの関係、ScalarDB/ScalarDLが配置される位置、協調データベース（Coordinator DB）の有無など。過度に詳細にせず、提案時に説明できる粒度でまとめる。 |
| **導入メリット・ROI** | 導入による具体的なメリット。データ統合による開発工数削減や運用コスト削減、マイクロサービス間の整合性担保による障害削減、リアルタイム分析とAI適用による新しい価値創出、ベンダーロックイン回避など。 |
| **類似事例／参考資料** | 同様のユースケースや事例があればリンクや資料名を記載する。顧客が検討する際に参考となる。 |

## 補足事項

### 産業・顧客属性の分類

産業や事業規模ごとにユースケースを分類することで、提案時に適した事例を紹介できる。例えば「メインフレームモダナイゼーション（金融・保険）」「IoTデータトラッキング（エネルギー）」「AI/LLMデータ基盤（製造）」など。

### ScalarDBの優位性

提案の際にはScalarDBの独自性を示すことが重要。複数データベースを横断したトランザクションと分析を同時に実現する点や、分散トランザクションのACID保証、データベース非依存性、ベンダーロックイン回避、統一セキュリティ、RAG対応などを強調する。

### 課題とソリューションの紐づけ

各ユースケースでビジネス課題・システム課題に対してどの機能や構成で解決するのかを明確に紐づける。課題が明確でない場合はヒアリングシートにして、顧客から情報を引き出す。

## まとめ

これらの情報を体系的に整理することで、顧客から課題を聞いたときに、適切なユースケースや構成を迅速に提案できるようになります。
